# 微光港湾（Weilight Harbor）— 开发问答

---

## 一、写代码过程中遇到了什么挑战？是怎么解决的？

### 挑战 1：敏感词检测系统的设计与性能优化

**问题描述：**
社区是面向重病患者家属的情感支持平台，用户可能在帖子中表达极度负面情绪甚至自伤倾向。我们需要同时实现两种完全不同的内容审核逻辑：
- **危机级别**（如"自杀""不想活"等自伤关键词）：需要**立即弹出危机热线弹窗**，给用户提供心理援助资源，而不是简单地屏蔽内容。
- **普通敏感词**（如辱骂、诈骗相关词汇）：需要将帖子**自动隐藏为待审核状态**，等管理员审核后再决定是否发布。

同时，敏感词检测在多个功能模块中被调用（社区发帖、评论、深夜独白、实时输入检测 API），如果每次都查询数据库，会造成性能瓶颈。

**解决方案：**

1. **分级严重度模型**：在数据库中为每个敏感词设置 `severity` 字段（1=警告级，2=危机级）。检测逻辑根据严重度返回不同响应：
   - severity ≥ 2 → 前端弹出包含心理援助热线的温暖弹窗，不提交内容
   - severity ≥ 1 → 帖子以 `status=0`（隐藏）状态入库，等待管理员审核
   
2. **内存缓存机制**：实现了带 5 分钟 TTL（生存时间）的内存缓存，敏感词列表从数据库懒加载后缓存在内存中，避免每次请求都查询数据库。缓存过期后自动重新加载。

3. **前端实时检测**：提供了 `/community/check-sensitive` API 接口，前端使用 800ms 防抖（debounce）机制在用户输入时实时调用，实现"边输入边检测"的体验，同时避免频繁请求服务器。

4. **差异化处理**：不同模块对检测结果的处理方式不同：
   - 发帖：隐藏待审核（软处理，不丢失用户内容）
   - 评论：直接阻止提交（严格处理）
   - 深夜独白：危机词返回 JSON 标记，前端显示关怀弹窗

### 挑战 2：深夜独白功能的时间限制与匿名机制

**问题描述：**
深夜独白（Night Confessions）是一个特色功能——只在晚上 21:00 到凌晨 6:00 开放，完全匿名，让用户在深夜说出白天不敢说的话。实现这个功能遇到了几个技术难点：
- 如何在前端和后端同时控制时间窗口？
- 匿名用户没有登录账号，如何追踪同一个用户的多条独白？
- 如何生成不重复且有温度的匿名昵称？

**解决方案：**

1. **双层时间验证**：
   - 前端：页面加载时检测当前时间，非开放时段显示倒计时和"月亮休息中"的提示动画
   - 后端：`POST` 接口再次验证服务器时间（UTC 21:00-06:00），防止用户通过修改本地时间绕过限制

2. **Session ID 追踪**：为未登录用户生成唯一的 session ID，存储在浏览器 session 中。这样即使用户没有登录，系统也能识别同一个浏览器会话发出的多条独白，为后续功能（如限流）提供基础。

3. **随机动物昵称**：预设了一组有温度的匿名昵称（萤火虫 Firefly、星雀 Starling 等），每条独白随机分配一个。虽然理论上存在昵称碰撞的可能，但在当前用户量级下概率极低，且碰撞本身不影响功能。

### 挑战 3：喘息驿站的地图交互与移动端适配

**问题描述：**
喘息驿站需要在地图上展示求助请求和器械出借信息，用户可以按距离筛选、点击标记查看详情、发布和接单。这在桌面端用 Leaflet.js + 侧栏列表的方案比较成熟，但在移动端小屏幕上同时展示地图和列表非常困难。

**解决方案：**

1. **响应式双布局**：
   - 桌面端：左侧列表（可收起/展开）+ 右侧地图的经典分栏布局
   - 移动端：全屏地图 + 底部弹窗（bottom sheet）替代侧栏，用户上滑查看列表

2. **标记聚合**：使用 `Leaflet.markercluster` 插件，在缩小地图时自动将密集的标记聚合为数字气泡，避免标记重叠导致的视觉混乱。

3. **颜色编码与脉冲动画**：橙色标记=求助服务，绿色标记=器械出借；待响应的请求有脉冲动画效果，让用户一眼区分紧急程度。

4. **半径预设**：提供 3km/5km/10km/20km 快捷按钮，点击后地图自动缩放到对应范围，比手动缩放更直观。

### 挑战 4：管理后台与前台系统的整合

**问题描述：**
我们的管理后台基于 `pear-admin-flask`（一个基于 Layui 的 Flask 管理后台框架），但前台是用 Bootstrap 5 + 自定义暖色主题构建的。两套 UI 框架并存带来了：
- 路由命名空间冲突
- 模板继承链不同（Layui layout vs Bootstrap layout）
- 用户认证系统需要同时服务前台普通用户和后台管理员

**解决方案：**

1. **蓝图（Blueprint）隔离**：将管理后台的认证审核、筹款审核、社区审核分别注册为独立的 Flask Blueprint，URL 统一以 `/admin/` 开头，与前台路由完全隔离。

2. **权限装饰器**：使用 `@authorize("权限代码", log=True)` 装饰器保护所有管理后台路由，基于 RBAC（基于角色的访问控制）模型，只有管理员角色才能访问后台功能。

3. **双模板目录**：前台模板放在 `templates/` 下使用 Bootstrap 布局，管理后台模板放在 `templates/admin/` 下使用 Layui 布局，通过不同的 `{% extends %}` 基类彻底分离。

4. **统一用户模型**：前台用户和管理员共用同一个 `User` 模型，通过 `role` 关联表区分权限，避免维护两套用户表。

## 二、敏感词检测具体有哪些词？

敏感词分为两个级别，共 **15 个词**：

### 危机级别（Severity 2）—— 触发心理援助弹窗

| 中文 | 英文 |
|------|------|
| 自杀 | suicide |
| 不想活 | kill myself |
| 结束生命 | end my life |
| 活不下去 | want to die |
| 轻生 | no reason to live |

> 检测到这些词时，前端会弹出温暖的危机干预弹窗，显示心理援助热线号码，内容**不会被提交**。

### 警告级别（Severity 1）—— 帖子自动隐藏待审核

| 中文 | 英文 |
|------|------|
| 骗子 | scam |
| 诈骗 | fraud |
| 傻逼 | damn |
| — | stupid |
| — | idiot |

> 检测到这些词时，帖子会以隐藏状态（`status=0`）保存到数据库，等待管理员在后台审核后决定是否公开。

### 检测机制说明

- **匹配方式**：大小写不敏感的子字符串匹配（如输入"SUICIDE"也会被检测到）
- **检测范围**：标题 + 正文合并检测
- **缓存策略**：词库从数据库加载后缓存 5 分钟，避免频繁查询
- **管理方式**：目前通过 `flask seed` 命令初始化词库，后续可在管理后台扩展管理界面

## 四、敏感词要怎么测试？

### 方法 1：在社区发帖测试（最直观）

1. 登录任意账号（如 `limei` / `test123`）
2. 进入**共鸣社区**页面，点击"发帖"
3. 在帖子内容中输入包含敏感词的文字：

**测试危机级别（弹窗效果）：**
- 输入"我不想活了"或"suicide"
- 预期结果：弹出温暖的危机干预弹窗，显示心理援助热线，内容不会被提交

**测试警告级别（隐藏待审核）：**
- 输入"这个骗子"或"scam"
- 预期结果：帖子成功提交，但提示"内容待审核"，帖子以隐藏状态存入数据库

4. 登录管理后台（`admin` / `admin123`），进入**社区审核**模块，可以看到被拦截的帖子，管理员可以选择通过或删除。

### 方法 2：在深夜独白测试

1. 在晚上 21:00 之后访问共鸣社区页面（右侧深夜独白区域会亮起）
2. 输入包含危机关键词的内容（如"我想结束生命"）
3. 预期结果：弹出关怀弹窗，显示支持信息和热线号码

### 方法 3：实时输入检测（API 测试）

在社区发帖的输入框中慢慢输入敏感词，输入停顿约 800ms 后，系统会通过 AJAX 调用 `/community/check-sensitive` 接口进行实时检测，如果是危机级别词汇，会立即弹出提示。

### 方法 4：通过命令行测试（开发者）

```bash
# 进入项目目录
cd weilight-harbor

# 激活虚拟环境
source venv/bin/activate

# 进入 Flask shell
flask shell

# 测试敏感词检测
from applications.common.utils.sensitive import check_sensitive, has_crisis_words

# 测试普通敏感词
print(check_sensitive("这个骗子太可恶了"))
# 输出: [{'word': '骗子', 'severity': 1}]

# 测试危机词汇
print(check_sensitive("我不想活了"))
# 输出: [{'word': '不想活', 'severity': 2}]

# 检查是否包含危机词
print(has_crisis_words("我想自杀"))
# 输出: True

# 正常内容
print(check_sensitive("今天天气真好"))
# 输出: []
```

### 完整测试流程建议

1. **先测试正常发帖**：发一条不含敏感词的帖子 → 应立即可见
2. **测试警告级敏感词**：发一条包含"骗子"的帖子 → 应提示待审核
3. **测试危机级敏感词**：输入"不想活" → 应弹出危机热线弹窗
4. **管理后台审核**：登录 admin 后台 → 社区审核 → 可以看到被拦截的帖子 → 通过/隐藏/删除
5. **通过后验证**：管理员通过帖子后 → 回到社区页面 → 帖子变为可见
